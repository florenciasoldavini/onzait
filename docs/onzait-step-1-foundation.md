# Onzait Design System

## Step 1 Foundation

This document defines the strategic foundation for the Onzait design system.

Its job is to answer the questions that should be decided before colors, components, and patterns become formalized:

- What is Onzait trying to help people do?
- Who are we designing for?
- How should the product feel in use?
- What principles should guide interface decisions?
- What should the system intentionally avoid?
- How will we know the system is working?

This is the decision layer for the design system. Tokens, components, and templates should follow this foundation.

## Product Purpose

Onzait helps construction and architecture teams coordinate the operational side of project work with more clarity, trust, and structure.

The product should support the day-to-day realities of delivery:

- tracking work
- coordinating teams
- communicating updates
- maintaining project visibility
- reducing ambiguity between field, office, and client

Onzait is not a marketing experience disguised as a productivity tool. It is a working environment for real project operations.

## Primary Users

The system should primarily serve professionals who are responsible for precision, coordination, and accountability.

Core user groups:

- project managers
- architects
- site supervisors
- crew leads
- operations coordinators
- clients reviewing progress and status

These users are busy, detail-oriented, and outcome-driven. They do not need playful UI. They need an interface that feels reliable, legible, and controlled.

## Product Personality

Onzait should feel:

- precise
- calm
- structured
- trustworthy
- modern
- premium
- technically literate

Onzait should not feel:

- cute
- casual
- noisy
- gimmicky
- decorative for its own sake
- generic startup SaaS

The product should feel like a digital operations layer built with architectural discipline.

## Core UX Principles

Every major UI decision should be tested against these principles.

### 1. Clarity Over Decoration

The interface should communicate hierarchy, state, and action clearly before trying to impress visually.

If a design decision looks good but slows comprehension, clarity wins.

### 2. Structure Over Spontaneity

Layouts should feel engineered, not improvised.

Spacing, alignment, borders, and grouping should create a sense of intentional order.

### 3. Trust Over Novelty

Users should feel that the product is stable, serious, and dependable.

Motion, color, and visual accents should support confidence, not perform for attention.

### 4. Calm Over Noise

The interface should reduce cognitive load.

Avoid excessive visual competition, too many accents, unnecessary labels, or over-styled containers.

### 5. Fast Comprehension Over Feature Density

Users should understand where they are, what matters, and what to do next in seconds.

Information-rich screens are acceptable, but they must still scan quickly.

### 6. Reuse Over Reinvention

Repeated problems should be solved in repeated ways.

If two screens need the same interaction pattern, they should not invent two visual languages.

## Interface Principles

These principles translate the brand into concrete UI behavior.

- Primary actions should be obvious and singular.
- Secondary actions should support the flow without competing with the main task.
- Forms should feel secure, legible, and calm.
- Dense information should be structured with rhythm, not compressed into clutter.
- Empty space should be used intentionally to create hierarchy and focus.
- Typography should do most of the hierarchy work before color or decoration does.
- Mono accents should signal technical metadata, not dominate the screen.
- Visual styling should support operational seriousness without feeling cold or hostile.

## Anti-Principles

The design system should explicitly avoid the following patterns:

- generic gradient-heavy SaaS aesthetics
- overly soft or bubbly interfaces
- playful UI patterns that weaken trust
- decorative motion without functional meaning
- too many accent colors competing for attention
- oversized cards and containers that add softness without clarity
- inconsistent spacing from screen to screen
- hardcoded one-off styles that bypass the system
- duplicated components that differ only slightly

## Accessibility Baseline

Accessibility is not a later polish step. It is part of the system foundation.

Onzait should aim for:

- strong contrast between text and background
- clear focus states
- readable type sizes
- meaningful labels for fields and actions
- touch targets that are easy to use on mobile
- hierarchy that does not rely on color alone

A premium interface that is hard to read or operate is not premium.

## Content and Tone Principles

Product language should match the same values as the interface.

Copy should be:

- direct
- calm
- concise
- competent
- human without being overly casual

Avoid:

- hype language
- overly cheerful system messages
- vague labels
- clever wording that hides meaning

When in doubt, choose the version that is clearer and more operationally useful.

## Success Criteria

The design system is succeeding when:

- new screens feel like part of the same product without feeling repetitive
- users can scan forms and dashboards quickly
- primary actions are obvious without excessive styling
- dense information remains readable and structured
- shared patterns reduce one-off design decisions
- new features can be built from system parts instead of starting from scratch
- the interface feels more trustworthy as the product grows, not less

## Decision Filter

When evaluating a new UI decision, ask:

1. Does this improve clarity?
2. Does this reinforce structure?
3. Does this increase trust?
4. Does this reduce noise?
5. Can this be reused consistently?

If the answer is no to most of these, it probably does not belong in the system.

## Practical Outcome

Step 1 is complete when the team can use this document to judge future design decisions before touching code or Figma.

This foundation should guide:

- token creation
- component design
- layout systems
- content patterns
- interaction rules
- future design reviews
